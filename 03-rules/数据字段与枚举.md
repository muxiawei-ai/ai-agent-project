# 数据字段与枚举

各工作表均含 `demo_flag` 列（`YES` 表示虚构演示数据，正式使用前删除）。

人工录入的枚举列均已配置引用 `08_Lookups` 的下拉数据验证；新增行请沿用下拉选择，不要手工输入枚举值。`99_Selfcheck` 工作表用演示数据校验判分公式，改动模板后必须全部通过。

## 1. `01_KB_Catalog`

| 字段 | 说明 |
|---|---|
| `qa_id` | 稳定唯一问答ID |
| `canonical_question` | 标准问题 |
| `answer_text` | 批准答案原文 |
| `intent_name` | 意图名称 |
| `scope_in` / `scope_out` | 适用/不适用范围 |
| `roles_applicable` | 适用角色 |
| `stages_applicable` | 适用阶段 |
| `preconditions` | 前置条件 |
| `risk_level` | 风险等级 |
| `compliance_sensitive` | 是否合规敏感 |
| `confusable_with` | 易混问答ID |
| `duplicate_answer_group` | 相同答案文本分组 |
| `normalized_answer` | 公式辅助字段（自动计算，勿手填） |
| `duplicate_answer_count` | 公式辅助字段（自动计算，勿手填） |
| `kb_version` | 知识库版本 |
| `demo_flag` | 演示数据标记 |
| `review_reason` | 模型不确定判断说明（来自 Prompt 00 输出） |
| `generator_run_id` | 生成批次（来自 Prompt 00 输出） |
| `generator_config` | 模型、Prompt版本、知识库版本（来自 Prompt 00 输出） |

导入映射：Prompt 00 输出的 `kb_catalog.jsonl` 按字段名对应填入本表；`normalized_answer`、`duplicate_answer_count` 为工作表公式列，`demo_flag` 由人工维护，均不在 JSONL 中。

本表用于判分反查时，只能包含当前冻结版本的答案（见判分规则第1节）。

## 2. `02_KB_Audit`

`finding_id,qa_id,finding_type,severity,description,suggested_action,business_decision,business_approval,compliance_approval,review_note,demo_flag`

## 3. `03_Candidate_Cases`

`case_id,suite_type,test_family,input_text,source_qa_id,variant_tags,role,stage,risk_level,model_match_candidate,coverage_decision,suggested_action,suggested_answer_draft,needs_review_reason,business_decision,approved_route,approved_qa_id,business_approval,compliance_approval,review_note,generator_run_id,generator_config,demo_flag`

## 4. `03b_Review_Results`（Prompt 04 独立复核输出）

`case_id,review_recommendation,duplicate_cluster,recommended_representative,recommended_test_family,recommended_coverage_decision,reason,evidence_qa_ids,missing_review_flag,demo_flag`

## 5. `04_Frozen_Cases`

`case_id,suite_type,test_family,kb_version,Q,expected_route,expected_qa_id,expected_answer_text,acceptable_answer_set,risk_level,business_approval,compliance_approval,demo_flag,freeze_gate`

`freeze_gate` 为公式列：审批硬门满足时为 `OK`，否则为 `NOT_APPROVED` 并在判分中计为 P0。

## 6. `05_Run_Results`

`run_id,case_id,environment_id,kb_version,returned_text,repeat_index,execution_note,remediation_owner,remediation_status,demo_flag`

执行人只填写前六列和 `demo_flag`；`remediation_owner`、`remediation_status` 在判分完成、确认失败归属后再填写。

## 7. `06_Scoring` 派生字段

全部为公式列，禁止手工修改：

| 字段 | 说明 |
|---|---|
| `Normalized_Actual` / `Normalized_Expected` | 标准化文本 |
| `Actual_Route` | 反查路由：`QA/BUSINESS_FALLBACK/HUMAN_HANDOFF/UNKNOWN/ERROR` |
| `Actual_QA_ID` | 唯一反查ID或 `MULTIPLE_IDS_SAME_TEXT` |
| `Pass_Fail` | `PASS/FAIL/BLOCKED` |
| `Failure_Type` | 见失败枚举 |
| `Is_P0` | `YES/NO` |
| `Consistency_Flag` | `SINGLE_RUN/CONSISTENT/INCONSISTENT`（同一 `run_id` 内比较） |
| `Version_Match` | `OK/VERSION_MISMATCH`；执行 `kb_version` 与冻结版本比对，不一致判 `BLOCKED` |

## 8. 枚举

### `suite_type`

`WORKING,BLIND`

### `test_family`

`POSITIVE,CONFUSION,GAP,SAFETY,ROBUSTNESS`

### `risk_level`

`LOW,MEDIUM,HIGH,CRITICAL`

### `coverage_decision`

`MATCH_EXISTING_QA,EXTEND_EXISTING_QA,ADD_NEW_QA,BUSINESS_FALLBACK,HUMAN_HANDOFF,COMPLIANCE_REVIEW,PRODUCT_RULE_GAP,INVALID_CASE`

### `approval`

`PENDING,APPROVED,REJECTED,NOT_REQUIRED`

### `route`

`QA,BUSINESS_FALLBACK,HUMAN_HANDOFF`

### `pass_fail`

`PASS,FAIL,BLOCKED`

### `failure_type`

`WRONG_ROUTE,FALSE_FALLBACK,FORCED_ANSWER,WRONG_FALLBACK_TIER,FALLBACK_TEXT_MISMATCH,UNKNOWN_OUTPUT,EMPTY_OR_ERROR,CASE_NOT_FOUND,VERSION_MISMATCH`

重复执行不一致（`INCONSISTENT`）由 `Consistency_Flag` 列单独表达，不出现在 `failure_type` 中。

### `review_recommendation`（独立复核）

`KEEP,MERGE,RECLASSIFY,REJECT,REVIEW`

### `remediation_owner`

`BUSINESS,COMPLIANCE,PRODUCT,IT,TEST`

### `remediation_status`

`OPEN,IN_PROGRESS,RESOLVED,ACCEPTED_RISK,NOT_REQUIRED`
