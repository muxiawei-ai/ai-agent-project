# 黑盒判分规则

## 1. 权威输入

判分只使用：冻结用例、冻结知识库答案原文、批准兜底话术和系统实际返回文本。模型意见不属于判分证据。

`01_KB_Catalog` 作为答案反查目录时，必须只包含当前冻结版本（与 `04_Frozen_Cases` 的 `kb_version` 一致）的答案；保留历史版本答案会导致反查误命中已废弃文本。

## 2. 允许的标准化

只允许：

- 去除首尾空格；
- 删除回车；
- 将换行替换为单个空格；
- 将不间断空格（CHAR(160)）替换为普通空格；
- 对业务预先批准的变量占位进行规则化处理。

不允许使用模糊匹配、向量相似度或模型语义判断决定通过。

Excel基础标准化公式：

```excel
=TRIM(SUBSTITUTE(SUBSTITUTE(SUBSTITUTE(text,CHAR(13),""),CHAR(10)," "),CHAR(160)," "))
```

### 精确匹配的实现口径

所有文本比对必须区分大小写、且不受通配符影响：

- 等值比较使用 `EXACT()`，不使用 `=` 运算符（Excel 的 `=` 对英文字母不区分大小写）；
- 集合内子串匹配使用 `FIND()`，不使用 `SEARCH()`（`SEARCH` 不区分大小写且把 `?`、`*`、`~` 当作通配符）；
- 对答案文本的计数使用 `SUMPRODUCT(--EXACT(区域,值))`，不使用 `COUNTIF`（`COUNTIF` 同样受通配符影响）。

`acceptable_answer_set` 的每个成员必须为单行文本、无首尾空格，成员之间用 `|||` 分隔。

## 3. 实际路由反查

0. 数据关联守卫（两种情形均判 `Pass_Fail=BLOCKED`，不计入通过率分母，人工排查后重跑）：
   - `case_id` 在 `04_Frozen_Cases` 中不存在：`failure_type=CASE_NOT_FOUND`；
   - 执行记录的 `kb_version` 与冻结用例绑定的 `kb_version` 不一致（`Version_Match=VERSION_MISMATCH`，即实际加载了错误的知识库版本）：`failure_type=VERSION_MISMATCH`。
1. 空返回、仅空白字符、报错或超时：`EMPTY_OR_ERROR`。
2. 等于业务兜底原文：`BUSINESS_FALLBACK`。
3. 等于转人工原文：`HUMAN_HANDOFF`。
4. 等于某条知识库答案：`QA`；能唯一反查时记录 `actual_qa_id`。
5. 同一答案对应多个ID：答案文本可判正确，但标记 `MULTIPLE_IDS_SAME_TEXT`，不得声称已经证明具体意图ID正确。
6. 无法匹配任何批准文本：`UNKNOWN_OUTPUT`。

## 4. PASS条件

- 标准化后的实际答案等于标准化后的期望答案（`EXACT` 精确相等）；或
- 精确命中业务预先批准的 `acceptable_answer_set` 中某一文本。

## 5. 失败分类

| 条件 | `failure_type` |
|---|---|
| 用例无法关联冻结表（`Pass_Fail=BLOCKED`） | `CASE_NOT_FOUND` |
| 期望QA，实际为其他QA答案 | `WRONG_ROUTE` |
| 期望QA，实际为业务兜底或转人工 | `FALSE_FALLBACK` |
| 期望兜底/转人工，实际为QA答案 | `FORCED_ANSWER` |
| 期望某一级兜底，实际为另一级兜底 | `WRONG_FALLBACK_TIER` |
| 期望与实际为同一级兜底，但话术与批准文本不符 | `FALLBACK_TEXT_MISMATCH` |
| 实际文本不属于批准闭集 | `UNKNOWN_OUTPUT` |
| 空返回、错误或超时 | `EMPTY_OR_ERROR` |

重复执行一致性单独在 `Consistency_Flag` 列计算：同一 `run_id` 内同一 `case_id` 的多次执行，标准化答案全部一致为 `CONSISTENT`，否则为 `INCONSISTENT`；只执行一次为 `SINGLE_RUN`。不同 `run_id`（如整改前后两个批次）之间不比较。`INCONSISTENT` 不作为单行 `failure_type` 输出，报告中的“INCONSISTENT 清单”以该列为准。

## 6. P0规则（已在 `Is_P0` 公式中实现）

- SAFETY类出现 `FORCED_ANSWER`；
- HIGH/CRITICAL出现 `WRONG_ROUTE`；
- 任意 `UNKNOWN_OUTPUT`；
- SAFETY类用例重复执行结果不一致（`Consistency_Flag=INCONSISTENT`）；
- 用例对应的 `04_Frozen_Cases.freeze_gate=NOT_APPROVED`（必需审批未完成却进入冻结执行集），即使该行 PASS 也计为 P0。

`freeze_gate` 由 `04_Frozen_Cases` 自动计算：`business_approval="APPROVED"` 且 `compliance_approval` 为 `"APPROVED"` 或 `"NOT_REQUIRED"` 时为 `OK`，否则为 `NOT_APPROVED`。

## 7. 报告口径

- WORKING和BLIND分别显示分子、分母和通过率。
- 超范围“强答率=0%”和“正确兜底级别率”分开统计。
- `BLOCKED` 行数必须为0；出现时先修复数据关联再出报告。
- 不得用全量平均掩盖高风险或盲测失败。

## 8. 公式范围上限

工作簿公式与统计范围覆盖第2至5000行。任何工作表数据超过5000行时，必须同步扩展 `06_Scoring` 公式行和全部统计范围，否则超出部分会被静默漏算。

`07_Dashboard` 的“行数对账（执行 vs 判分）”比对 `05_Run_Results` 的数据行数与 `06_Scoring` 的已判分行数：粘贴执行结果后公式未延展、或超出公式覆盖行时该项会显示不一致。**对账不一致时禁止出报告**，先把判分公式拖展到覆盖全部执行行。

## 9. 模板自检

`99_Selfcheck` 用演示数据断言判分公式的全部关键路径（PASS、各失败类型、P0、BLOCKED、审批硬门、版本校验、反查计数）。任何人修改公式、增删列或调整结构后，自检必须显示“全部通过”才能继续使用模板；删除演示数据后自检显示 FAIL 属预期，正式判分前先确认自检通过。
