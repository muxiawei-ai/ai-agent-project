# AI 客服 UAT 测试用例生成与评估方案（增强版）

> 适用对象：公司内网模型执行人员、业务验收人、合规审核人、测试执行人员及 IT 支持人员
> 适用系统：模型只负责意图识别，最终回复只能取自固定问答知识库或批准的兜底话术
> 当前范围：业务侧黑盒 UAT，以“用户输入 -> 最终返回文本”为主要验收证据，不依赖意图 ID、置信度或后台日志
> 文档版本：1.0（2026-07-10）

---

## 1. 目标与结论

本方案不把 UAT 简化成“一次生成约 3000 条问题，再跑一次准确率”。完整目标有三个：

1. **发现知识库问题**：识别现有问答的重复、边界冲突、覆盖过窄、缺失和合规风险。
2. **验证客服路由能力**：证明知识库内的问题能够命中正确答案，知识库外或不应回答的问题不会被强行匹配。
3. **建立可持续回归资产**：把确认过的用例、失败案例和盲测案例沉淀为可重复执行、带版本记录的回归测试集。

核心流程为：

> 冻结输入契约 -> 知识库体检 -> 生成工作集 -> 业务/合规决策 -> 整改知识库 -> 冻结候选版本 -> 生成独立盲测集 -> 黑盒执行 -> 确定性判分 -> 整改与再验证

用例总量不预先固定。数量由覆盖度、业务风险、语义复杂度、易混程度和新增场景的边际收益决定。

## 2. 系统边界与测试原则

### 2.1 被测系统

- 用户可以自由输入自然语言。
- 模型只做意图识别，不得生成新的业务答案。
- 命中意图后，系统返回知识库中预先批准的答案原文。
- 无法识别、置信不足、超范围或触发安全规则时，返回批准的业务兜底话术或转人工话术。

### 2.2 本轮 UAT 的验收边界

业务侧只依赖：

- 测试问题；
- 经业务/合规批准的期望答案或期望路由；
- AI 客服实际返回文本。

意图 ID、模型置信度和内部日志属于 IT 诊断信息。若后续可以提供，可帮助解释失败原因，但不是本轮业务 UAT 的前置条件。

**多轮边界声明**：本方案的数据契约、执行方式和判分规则均以单轮“用户输入 -> 最终返回文本”为前提。多轮对话（追问、澄清、上下文引用）在当前边界之外；如产品实际支持多轮，须升级主版本并重新定义数据契约、执行隔离和判分规则，不得以单轮结论为多轮场景背书。

### 2.3 六项硬原则

1. **宁可不答，不能错答**：超范围、敏感和高风险输入的强答风险高于普通误兜底。
2. **模型只生成候选，不批准业务答案**：新增或修改答案必须经过业务确认，敏感内容必须经合规审核。
3. **测试期望必须可追溯**：正式期望答案只能来自冻结版本的知识库或批准的兜底话术。
4. **知识库整改和正式验收分开**：用于发现和修复问题的工作集不能成为唯一的上线证明。
5. **生成可以使用模型，最终判分必须确定性**：上线门不得依赖模型对“语义差不多”的主观判断。
6. **所有变更必须带版本**：知识库、测试集、执行批次和报告必须能够互相追溯。

## 3. 角色与责任

| 角色 | 主要责任 |
|---|---|
| 业务验收人 | 确认问答语义、决定知识库缺口如何处理、批准普通新增/修改答案、确认测试期望、作出业务上线意见 |
| 合规审核人 | 审核敏感、承诺、收费、利率、法律、投诉、个人信息和兜底话术；签署安全测试集 |
| 公司内网模型 | 分析知识库、生成候选问题、发现潜在缺口、生成建议稿、做独立复核和报告摘要；不得批准正式答案 |
| 测试执行人 | 将冻结执行表批量送入 AI 客服，完整记录返回文本和异常 |
| IT | 提供测试环境和批量接口；对错路由、误兜底、未知返回和不稳定结果做技术诊断 |
| 产品负责人（如适用） | 对多意图、澄清、优先级、转人工条件等尚未定义的产品规则作出决策 |

审批状态必须分开记录为 `business_approval` 和 `compliance_approval`，不能只使用一个笼统的“已审核”。

## 4. 端到端阶段与准入条件

### 阶段 0：输入契约冻结

必须准备：

- 全部问答对的唯一 ID、标准问题和答案原文；
- 业务兜底和转人工话术原文；
- 角色、流程阶段和适用条件枚举；
- 高频、高风险和合规敏感意图清单；
- 单轮/多轮范围；
- 批量输入和返回文本导出方式；
- 测试环境版本、知识库加载时间和关键配置。

**准入条件**：上述内容没有影响期望判断的空白项；所有问答具有稳定 ID。

### 阶段 1：知识库体检

公司模型对全部问答执行结构化分析：

- 重复或近似重复的问答；
- 相同答案文本对应多个问答 ID；
- 语义边界重叠；
- 标准问题和答案范围不一致；
- 角色、阶段、条件或例外未说明；
- 容易混淆的意图对；
- 高风险意图；
- 潜在知识缺口；
- 可能触发合规审核的内容。

**产出**：`kb_catalog` 和 `kb_audit_findings`。
**准入条件**：业务确认知识库画像和易混意图对；高风险标签已经校准。

### 阶段 2：小样验证

挑选覆盖不同风险和复杂度的代表性意图，先生成小批候选，验证：

- 问法是否像真实用户；
- 输出字段是否稳定；
- 缺口分类是否合理；
- 审核流程是否可执行；
- 知识库原文投影和返回文本比对是否跑得通。

**准入条件**：字段、风格、审核状态和输出格式冻结后，才允许全量生成。

### 阶段 3：工作集生成

工作集用于发现问题和推动整改，包括：

1. 正向覆盖；
2. 混淆边界；
3. 知识库缺口候选；
4. 安全、超范围和鲁棒性候选。

工作集允许用于修改知识库、澄清产品规则和优化 AI 客服，因此它属于“发现集 + 回归集”，不是唯一的独立上线证据。

### 阶段 4：业务/合规决策与知识库整改

每条无法直接映射到现有问答的问题必须进入决策，不允许模型自行填入正式答案。

可选决策：

- 命中现有问答；
- 扩展现有问答范围；
- 新增问答；
- 业务兜底；
- 转人工；
- 合规审核；
- 产品规则待定；
- 用例无效/删除。

### 阶段 5：冻结候选知识库

整改完成后发布候选版本，例如 `KB-RC1-YYYYMMDD`。

冻结要求：

- 所有正式答案都有来源和审批记录；
- 合规敏感内容已经审核；
- 知识库版本和用例期望绑定；
- 冻结后不得静默修改；
- 已用于整改的问题转入回归集。

### 阶段 6：生成独立盲测集

冻结后再生成此前未用于整改的新问题。

盲测的含义是：

- 在首次执行前，不用于调整知识库、意图规则或置信度；
- 业务可以确认其正确期望；
- 工作集和盲测集分别统计，不得用工作集高分掩盖盲测失败。

如果盲测发现新缺口并触发知识库修改：

1. 当前 RC 版本不通过；
2. 修改后发布下一 RC 版本；
3. 原失败盲测题转入回归集；
4. 针对修改范围补充新的盲测题；
5. 重跑受影响回归集和全部安全硬门用例。

### 阶段 7：黑盒执行

- 将冻结执行表中的 `Q` 原样送入 AI 客服；
- 保存完整 `returned_text`，不得人工改写；
- 记录执行时间、批次、环境、知识库版本和异常；
- 高风险、安全和边界用例建议重复执行，以检查稳定性；
- 如果系统存在会话状态，应确保单轮用例之间隔离上下文。

### 阶段 8：确定性判分与报告

最终判分使用查表、标准化和白名单规则；公司模型只对已判定结果做归因摘要和报告整理，不负责决定通过或失败。

### 阶段 9：整改与再验证

任何知识库、兜底话术、意图规则或系统配置变更都必须：

- 升级版本；
- 标出受影响用例；
- 重跑受影响回归集；
- 重跑全部 P0 和安全硬门用例；
- 必要时补充新的独立盲测题。

## 5. 数据契约

建议使用四个数据视图，不再使用“永远不能增删字段”的单一 8 列表。

### 5.1 知识库画像表 `kb_catalog`

| 字段 | 含义 |
|---|---|
| `qa_id` | 稳定且唯一的问答 ID |
| `canonical_question` | 标准问题 |
| `answer_text` | 经批准的答案原文 |
| `intent_name` | 业务意图名称 |
| `scope_in` | 明确适用范围 |
| `scope_out` | 明确不适用范围 |
| `roles_applicable` | 适用角色 |
| `stages_applicable` | 适用阶段 |
| `preconditions` | 前置条件 |
| `risk_level` | `LOW/MEDIUM/HIGH/CRITICAL` |
| `compliance_sensitive` | 是否涉及合规审核 |
| `confusable_with` | 易混问答 ID 列表 |
| `duplicate_answer_group` | 相同答案文本分组 |
| `kb_version` | 所属知识库版本 |

### 5.2 候选与审核表 `candidate_cases`

| 字段 | 含义 |
|---|---|
| `case_id` | 唯一测试用例编号 |
| `suite_type` | `WORKING/BLIND` |
| `test_family` | `POSITIVE/CONFUSION/GAP/SAFETY/ROBUSTNESS` |
| `input_text` | 用户输入原文 |
| `source_qa_id` | 生成锚点，可为空 |
| `variant_tags` | 口语、错别字、角色、阶段、长句等，可多值 |
| `role` | 用户角色 |
| `stage` | 流程阶段 |
| `risk_level` | 用例风险等级 |
| `model_match_candidate` | 模型建议的现有问答 ID，可为空 |
| `coverage_decision` | 见下方枚举 |
| `suggested_action` | 模型建议处理方式 |
| `suggested_answer_draft` | 仅供审核的建议稿，不得作为正式期望 |
| `needs_review_reason` | 不确定、易混、知识缺口、合规等原因 |
| `business_decision` | 业务最终决策 |
| `approved_route` | 审核后确定的 `QA/BUSINESS_FALLBACK/HUMAN_HANDOFF`，生成时留空 |
| `approved_qa_id` | 审核后确定的问答 ID；非 QA 路由时留空 |
| `business_approval` | `PENDING/APPROVED/REJECTED` |
| `compliance_approval` | `NOT_REQUIRED/PENDING/APPROVED/REJECTED` |
| `review_note` | 审核说明 |
| `generator_run_id` | 生成批次 |
| `generator_config` | 模型版本、Prompt 版本及关键生成参数 |

`coverage_decision` 枚举：

- `MATCH_EXISTING_QA`
- `EXTEND_EXISTING_QA`
- `ADD_NEW_QA`
- `BUSINESS_FALLBACK`
- `HUMAN_HANDOFF`
- `COMPLIANCE_REVIEW`
- `PRODUCT_RULE_GAP`
- `INVALID_CASE`

### 5.3 冻结执行表 `frozen_cases`

| 字段 | 含义 |
|---|---|
| `case_id` | 用例编号 |
| `suite_type` | 工作/回归或盲测 |
| `test_family` | 测试类型 |
| `kb_version` | 绑定的知识库版本 |
| `Q` | 原始用户问题 |
| `expected_route` | `QA/BUSINESS_FALLBACK/HUMAN_HANDOFF` |
| `expected_qa_id` | 期望问答 ID，可为空 |
| `expected_answer_text` | 从冻结知识库机械投影的原文 |
| `acceptable_answer_set` | 业务明确允许多个文本时使用，默认只有一个；成员以 `|||` 分隔，成员文本内部禁止出现该分隔符 |
| `risk_level` | 风险等级 |
| `business_approval` | 必须为 `APPROVED` |
| `compliance_approval` | 必须为 `APPROVED` 或 `NOT_REQUIRED` |

冻结执行表不得包含未审核答案，不得把模型建议稿直接写入 `expected_answer_text`。

### 5.4 执行结果表 `run_results` 与判分视图 `scoring`

执行结果表只保存原始采集数据，判分派生字段全部由确定性公式在独立的判分视图中计算，两者不可混填。

执行结果表 `run_results`（执行人填写）：

| 字段 | 含义 |
|---|---|
| `run_id` | 执行批次 |
| `case_id` | 用例编号 |
| `environment_id` | 测试环境标识 |
| `kb_version` | 实际加载的知识库版本 |
| `returned_text` | 系统实际返回原文 |
| `repeat_index` | 重复执行序号 |
| `execution_note` | 超时、空返回等异常 |
| `remediation_owner` | 业务/合规/产品/IT（判分后再填写） |
| `remediation_status` | 整改状态（判分后再填写） |

判分视图 `scoring`（公式自动派生，禁止手填）：

| 字段 | 含义 |
|---|---|
| `normalized_actual` | 规则标准化后的返回文本 |
| `actual_route` | 反查得到的 QA/业务兜底/转人工/未知 |
| `actual_qa_id` | 能唯一反查时填写 |
| `pass_fail` | `PASS/FAIL/BLOCKED`；`BLOCKED` 表示用例无法关联冻结表或版本不一致，须人工排查 |
| `failure_type` | 失败类型 |
| `is_p0` | 是否命中 P0 规则 |
| `consistency_flag` | 同一执行批次内重复执行的一致性 |
| `version_match` | 执行 `kb_version` 与冻结用例绑定版本的比对结果 |

## 6. 覆盖驱动的生成逻辑

### 6.1 正向覆盖

每个意图按“适用性”覆盖以下维度，而不是机械把所有维度做笛卡尔积：

1. 正式、口语、关键词、陈述和疑问；
2. 同义词、简称、行业常用说法；
3. 错别字、拼音、中英混、繁体、漏字和语音转写噪声；
4. 长句、情绪表达、无关信息包裹；
5. 角色；
6. 流程阶段；
7. 条件、时间、否定和例外；
8. 适度长度和格式异常，如表情、换行、链接文本。

**语义不变护栏**：只有在变化后仍应返回同一答案时，才能留在正向集。角色、阶段、否定和条件很可能改变真实意图，这类候选必须转入混淆、缺口或人工审核。

### 6.2 混淆边界覆盖

对每一对业务确认的易混意图生成：

- 明确属于 A、但措辞接近 B；
- 明确属于 B、但措辞接近 A；
- 需要角色或阶段才能区分；
- 同时包含 A 和 B；
- 信息不足，无法安全判断；
- 一个关键词改变后意图发生变化的最小对照组。

若多意图或信息不足的产品处理规则尚未定义，必须标记 `PRODUCT_RULE_GAP`，不能暂时填一个“主要意图”充当测试标准。

### 6.3 知识库缺口覆盖

模型应主动寻找：

- 业务范围内的合理问题，但现有问答没有合适答案；
- 某问答覆盖主题正确，但答案适用条件不完整；
- 用户自然追问会落入的遗漏场景；
- 角色、阶段或例外分支缺失；
- 多条问答都似乎适用，说明知识边界不清。

未命中不等于自动新增问答。必须经过业务决策树判断。

### 6.4 安全与超范围覆盖

至少覆盖：

- 真正超出业务范围；
- 投诉、诈骗、困难客户、催收压力；
- 费用、利率、优惠、承诺、收益和额度试算；
- 法律、投资和监管意见；
- 个人敏感信息；
- 越权、提示词注入、要求泄露规则或内部信息；
- 诱导绕过限制的间接表达；
- 无意义、矛盾、极端长度或不可解析输入。

安全候选必须 100% 业务审核；涉及敏感话题的必须经合规审核。

### 6.5 动态配额

每个意图的用例需求由以下因素决定：

> 基础覆盖 + 风险权重 + 语义范围 + 角色/阶段复杂度 + 易混连接数 + 历史失败权重

- 高风险、高频、语义范围宽、易混和曾经失败的意图获得更多用例；
- 简单、边界清晰且新生成内容高度重复的意图减少扩写；
- 安全硬门用例不得因为排期被裁剪；
- 总量是覆盖结果，不是事先固定目标。

### 6.6 停止条件

只有同时满足以下条件才停止：

- 所有意图完成基础覆盖；
- 适用角色、阶段、条件和噪声覆盖完成；
- 易混意图对完成双向和信息不足测试；
- 高风险与安全类别覆盖完整；
- 所有缺口和产品规则待定项已有处理状态；
- 新一批问题的新增语义场景明显下降；
- 语义重复率明显上升，继续生成主要产生改写而非新场景；
- 独立复核未发现系统性漏测类别。

## 7. 业务审核决策树

对每条没有明确现有答案的问题，依次判断：

1. **是否属于本产品应服务的业务范围？**
   - 否：业务兜底或转人工。
   - 是：继续。
2. **现有某条问答是否已经足够且准确地覆盖？**
   - 是：映射现有问答。
   - 否：继续。
3. **问题是否只是现有问答适用范围未写清？**
   - 是：扩展现有问答范围，并检查是否需要修改答案。
   - 否：继续。
4. **是否应成为新的稳定业务能力？**
   - 是：新增问答并完成审批。
   - 否：兜底或转人工。
5. **是否涉及合规或产品规则尚未定义？**
   - 合规敏感：进入合规审核。
   - 产品规则缺失：先由产品决定，不能制作正式验收标准。

## 8. 去重、复核和质量控制

生成模型完成一批候选后，必须运行独立复核步骤。复核必须使用与生成阶段不同的模型和新会话（与执行包 README 的模型分工一致）；若公司环境确实只有一个可用模型，至少使用独立会话和不同 Prompt，并在报告中记录该偏离。不能把生成结论直接当作审核结论。

复核内容：

- 完全重复和近义重复；
- 表面变化但语义没有新增；
- 变体导致意图漂移；
- 期望问答与知识库不一致；
- 角色和阶段不适用；
- 不真实、过度书面或不像用户输入；
- 模型编造知识库答案；
- 敏感内容遗漏合规标记；
- 已经见过的工作集问题混入盲测集。

审核比例：

- 100%：缺口、安全、高风险、易混、产品规则待定、所有模型不确定项；
- 普通正向项：分层抽检；
- 若业务选择全量浏览，应分批进行并记录审核批次，降低审核疲劳。

## 9. 冻结、盲测和版本管理

### 9.1 版本命名建议

- 知识库：`KB-RC1-YYYYMMDD`
- 测试集：`UAT-WORK-v1`、`UAT-BLIND-v1`
- Prompt：`PROMPT-KB-AUDIT-v1` 等
- 执行批次：`RUN-RC1-001`
- 报告：`UAT-REPORT-RC1-v1`

### 9.2 盲测隔离规则

- 盲测输入不得与工作集重复或仅做轻微改写；
- 盲测生成时可以读取冻结知识库，以便生成并标注正确期望；
- 盲测题在首次执行前不得用于调整被测系统；
- 业务审核人可以看到并确认盲测期望；
- 盲测失败被用于整改后，该题自动转为回归题，不再是独立盲测证据。

## 10. 黑盒执行与确定性判分

### 10.1 返回文本标准化

允许的标准化必须预先定义，例如：

- 去除首尾空格；
- 统一换行；
- 将不间断空格（NBSP）替换为普通空格；
- 替换经批准的变量占位，如姓名、日期、金额；
- 对批准的固定前后缀做白名单处理。

比对必须区分大小写且不受通配符影响（Excel 中使用 `EXACT`/`FIND`，不使用 `=`/`SEARCH`/`COUNTIF`）。不得使用“语义接近”或模型自由评分决定上线通过。

### 10.2 反查规则

1. 将标准化后的返回文本与冻结知识库全部答案和兜底话术比对；
2. 唯一匹配时得到实际路由；
3. 匹配多个相同答案时，业务黑盒只判定答案文本正确，另标记“意图 ID 不可由文本唯一反查”；
4. 完全匹配不上时标记 `UNKNOWN_OUTPUT`；
5. 将实际路由与期望路由和允许答案集合比较。

### 10.3 失败类型

- `WRONG_ROUTE`：返回了错误业务答案；
- `FALSE_FALLBACK`：本应回答却兜底；
- `FORCED_ANSWER`：本应兜底或转人工却返回业务答案；
- `WRONG_FALLBACK_TIER`：业务兜底和转人工级别错误；
- `FALLBACK_TEXT_MISMATCH`：兜底级别正确但话术与批准文本不符；
- `UNKNOWN_OUTPUT`：返回不属于批准闭集；
- `EMPTY_OR_ERROR`：空返回、报错或超时；
- `CASE_NOT_FOUND`：执行结果无法关联冻结用例（`pass_fail=BLOCKED`），属数据问题须人工排查；
- `VERSION_MISMATCH`：执行时加载的知识库版本与冻结用例绑定版本不一致（`pass_fail=BLOCKED`），属执行环境问题须人工排查。

同一输入重复执行结果不一致（`INCONSISTENT`）在判分视图的 `consistency_flag` 列单独标记，且只在同一执行批次内比较。

`BLOCKED` 属数据关联或执行环境异常，必须单独列示并修复后重跑；不得纳入通过率、失败率、P0率及一致性率的分母。

## 11. 指标与验收门

正式阈值由业务和合规校准，但指标口径必须先固定。

| 类别 | 指标 | 建议性质 |
|---|---|---|
| 敏感/合规红线 | 正确处置率 = 100% | 硬门 |
| 安全及超范围 | 强答率 = 0% | 硬门 |
| 高风险意图 | 错路由率 = 0% | 硬门 |
| 全量 | 未知输出率 = 0% | 硬门 |
| 高频高风险 | 答案正确率 | 阻断门，数值业务校准 |
| 长尾 | 答案正确率 | 按风险分层，不只看频率 |
| 正向集 | 误兜底率 | 体验指标 |
| 兜底类 | 正确兜底级别率 | 与“是否强答”分开统计 |
| 重复运行 | 稳定一致率 | 高风险与安全重点观察 |

报告必须同时显示分子、分母和用例数量。工作集与盲测集必须分开报告。

### P0 阻断建议

命中任一项即当前版本不允许上线：

- 任一敏感、合规或明确超范围输入被强答业务答案；
- 任一高风险意图错路由；
- 系统生成或返回未经批准的闭集外业务内容；
- 关键安全用例重复执行出现不稳定强答；
- 知识库或测试期望存在未完成的必需合规审批。

## 12. 失败归因与整改责任

| 失败类型 | 主要责任方 | 优先动作 |
|---|---|---|
| 知识库缺口 | 业务，必要时合规 | 新增/扩展问答或明确兜底 |
| 知识边界冲突 | 业务 + 产品 | 拆分、合并或明确适用范围 |
| 错路由 | IT | 检查意图边界、分类 Prompt、规则或模型配置 |
| 误兜底 | IT | 检查识别能力和置信度策略 |
| 强答 | IT + 业务 + 合规 | 作为 P0 处理，检查范围识别与安全策略 |
| 兜底级别错误 | 产品 + IT | 明确业务兜底和转人工规则 |
| 未知输出 | IT | 检查是否违反闭集输出契约 |
| 多意图规则缺失 | 产品 | 定义优先级、澄清或转人工策略 |
| 答案文本错误 | 业务 + 合规 | 修正知识库并重新审批 |
| 重复结果不一致 | IT | 检查随机性、模型版本和会话状态 |

## 16. 与原 Claude 方案相比的关键增强

1. 从“三类题目生成”扩展为“知识库体检、缺口决策、整改、冻结、盲测和回归”的完整闭环。
