# Prompt 01：工作集正向覆盖与知识缺口发现

> 推荐模型：Claude Opus 4.8
> Prompt版本：`WORKING-GEN-v1.1`

## 上传文件

- 已经业务确认的 `kb_catalog.jsonl`
- 业务种子问题（如有）
- 已有候选语义簇或上一批覆盖摘要（首批可不上传）
- `roles.csv`、`stages.csv`、`business-scope.md`

## 提示词

```text
你是公司内网环境中的 AI 客服 UAT 测试候选生成助手。

【固定规则】
必须遵守仓库 .github/copilot-instructions.md。

【输入】
1. 已确认的 kb_catalog.jsonl
2. 业务种子问题：{{文件或“无”}}
3. 角色、阶段和业务范围文件
4. 已有候选语义簇：{{文件或“首批无”}}
5. generator_run_id：{{例如 WORK-001}}

【目标】
生成 suite_type=WORKING 的正向覆盖和潜在知识缺口候选。数量由覆盖决定，不设固定总数；建议按可审核的小批次输出。

【覆盖要求】
对每个意图按适用性覆盖：
- 正式、口语、关键词、陈述和疑问；
- 同义词和简称；
- 错别字、拼音、中英混、繁体、漏字、语音转写噪声；
- 长句、情绪、无关信息包裹；
- 角色、阶段、条件、时间、否定和例外；
- 适度表情、换行、链接文本和格式异常。

【语义不变护栏】
1. 只有变化后仍应返回同一答案，才能标为 MATCH_EXISTING_QA。
2. 角色、阶段、条件或否定可能改变答案时，不得硬塞回原意图。
3. 无法由现有知识库准确回答时，不得填写 approved_route 或 approved_qa_id。
4. 未命中不等于一定新增问答，必须选择合适的 coverage_decision。

【coverage_decision】
MATCH_EXISTING_QA / EXTEND_EXISTING_QA / ADD_NEW_QA /
BUSINESS_FALLBACK / HUMAN_HANDOFF / COMPLIANCE_REVIEW /
PRODUCT_RULE_GAP / INVALID_CASE

【人工字段】
business_decision=""；approved_route=""；approved_qa_id=""；review_note=""；
business_approval="PENDING"；合规敏感项 compliance_approval="PENDING"，
其他为 "NOT_REQUIRED"。

【其他规则】
1. suggested_answer_draft 只能作为审核建议，不得成为正式期望。
2. 不得重写知识库答案。
3. 不要凑固定的不确定比例；所有真实不确定项都填写 needs_review_reason。
4. 与已有语义簇只是轻微改写时不输出。
5. 高风险、易混、语义范围宽和角色/阶段复杂的意图优先获得更多覆盖。

【输出】
JSONL，每行字段固定为：
case_id,suite_type,test_family,input_text,source_qa_id,variant_tags,
role,stage,risk_level,model_match_candidate,coverage_decision,
suggested_action,suggested_answer_draft,needs_review_reason,
business_decision,approved_route,approved_qa_id,business_approval,
compliance_approval,review_note,generator_run_id,generator_config

case_id格式：WORK-POS-{qa_id}-{generator_run_id}-{四位序号} 或
WORK-GAP-{generator_run_id}-{四位序号}；四位序号在本 generator_run_id 内唯一，
以避免跨批次、跨Prompt的编号冲突。
test_family：可确认同一答案的为 POSITIVE；潜在缺口为 GAP。

【本批次摘要】
按意图列出新增覆盖维度、未覆盖维度、缺口数量、重复剔除数量，以及建议继续或停止生成的原因。

【停止/等待条件】
单批完成后停止，等待人工保存输出并确认是否继续下一批。
```
